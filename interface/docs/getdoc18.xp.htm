Perhaps not.  I have gotten the PnP info without responding
to CM messages.  Here is what I posted earlier:
 
- - - - - - - - - - - - - - - - -
 
This is how I succeeded in finding a PnP device.  It may not
be the easiest way, so please correct me if I am making this
too complicated.
 
The device is a PCI-bus analog data acquisition board from
Computer Boards inc.  Their vendor number is Hex(1307)
and the device number of this particular model of board is
Hex(0019).  My VxD is dynamically loaded when my app starts
up.  As part of the Sys_Dynamic_Device_Init processing,
I search the Config Manager's DEVNODE tree using multiple
calls to _CONFIGMG_Locate_Devnode, _CONFIGMG_Get_Child,
and _CONFIGMG_Get_Sibling.  At each DEVNODE, I call
_CONFIGMG_Get_Device_ID, which returns a string.  I found
from trial and error that the string I want is
 
  PCI\VEN_1307&DEV_0019.... etc.
 
I suspect that the rest of the string is dependent on,
among other things, which PCI slot the board is in (in case
there are several boards of the same model, which is not a
problem for me since my app only uses one of these boards). 
So I stop my string comparison just after the "0019".  When I get a match, then I've found the board, so I can then query
for the base I/O addresses and the IRQ number.  One thing I noticed is that in Windows 98, the string after "0019" continues with
 
  &SUBSYS_00000000&REV_00
 
which is not present in the Windows 95 string.
 
I should mention that my VxD is not the "registered" driver for this board, but for reasons related to the limitations of
Computer Boards driver software, I needed to write my own.
Fortunately, their driver (which is statically loaded
from SYSTEM.INI) is benign inthat it does not claim
owership of the board or its IRQ when a call to their driver
is not in progress, thus allowing my VxD to work as if their
driver were not around.
 
So, under these conditions, is this the best/only way for
my VxD to get the PnP base addresses and IRQ number?
I'm not sure how to communicate directly with the PnP BIOS,
but I suspect that that way is even messier.
