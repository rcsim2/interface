"Rik G." <rgosens@zonnet.nl> wrote in message
news:u%H65.9132$vJ6.266600@zonnet-reader-1...
> >If the handle was opened with FILE_DELETE_ON_CLOSE
> >then the load count will be decremented for the
> >VXD; if the load count is now zero the VXD
> >will be requested to be unloaded.  The VXD
> >can deny the request.
 >
> What is it that makes a zero ref count vxd deny that request?
>
  
The VXD's programmer. I can't tell you what he had
in mind!
 
I can tell you why one of my VXDs might refuse
to unload: it's busy doing something.  (It might
have other handles open in addition to the
handle opened by, and now closed by, the process
which originally loaded the VXD.)
 
Note: the above may be surprising. It surprised
me. Windows will unload a VXD even though it
has open handles. The "ref count" you mention
isn't a ref count, it's a load count.
 
The mechanism by which a VXD refuses to unload is
to return 'false' from its dynamic-vxd-termination
message handler. That is, the OS says 'you are
being unloaded' and the VXD says 'no'.
 
(I'm only an amateur at these details. For more
knowledgeable info, seek out past postings by
intrepid Windows spelunker Geoff Chappell).
 
> Another interesting thing:
>
>      if ( DeleteFile("\\\\.\\VMYXD") ) {
>      printf("\nDeleteFile successful\n");
>      } else {
>      printf("\nDeleteFile failed\n");
>      }
>
> This always reports "DeleteFile successful" whether the vxd is loaded or
> not. Why is that?
  
DeleteFile is buggy. When unloading a VXD,
it returns 0 on success, non-zero on failure,
regardless of what the documentation says it
does.
