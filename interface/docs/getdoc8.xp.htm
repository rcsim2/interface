Subject:measure time between interrupts
Date:06/24/2000
Author: Peter Remmers <pitti@amendestr.in-berlin.de>

<< previous in search  ·  next in search >>

Hello!
 
My VxD has installed a hardware interrupt handler.
Now it needs to measure the time between interrupts, i.e. from
one interrupt to the next, at a resolution in the microseconds
range. The average time between interrupts is about 1.5ms.
 
Now I know that the services an interrupt handler is allowed to
call is restricted.
 
I know of three possibilities to measure the time:
 
- use the rdtsc instruction. But since this is CPU clock dependant   I would first have to measure how many CPU clocks are 1.5ms.
  Second, I don't know much about this instruction and where to fetch   the ticks after calling the instruction.
  Third and most important, I cannot rely on this instruction,
  because the CPU may not support it.
 
- use VTD_Get_Real_Time(). I don't know if an ISR is allowed to call   this service. My book says no (Thielen&Woodruff), at least it's   not in the book's list of services an ISR is allowed to call.
 
- read the system timer 0 with out 43h/in 40h/in 40h.
  This is the method I used in the DOS program I wrote.
  But I don't know if the timer is left at 18.2 Hz interrupt rate.   If windows changed the timer not to count from 10000h to zero
  but instead from a much lower value (increasing IRQ0 interrupt   rate), that may mean that I don't have enough time to stay
  within at most one counter wrap around.
  The times I may measure are in the range from 0.9ms to ~20ms.
 
Tests show that VTD_Get_Real_Time() works. But I don't know if
it does just work "most of the time".
What method do you recommend?
 
The times I measure jitter alot. Under pure DOS I get values
that jitter only about < +/- 5µs which is near perfect.
In my ISR the values jitter more than 10 times as much with
occasional *giant* glitches.
This is like running my DOS program in a DOS box.
I thought that a hardware ISR in a VxD has minimum latency
and I would get as perfect values as in pure DOS...
 
I hoped that it would not be necessary to have a lot expenditure to iron out the timing glitches like the DIGIJOY sample from the DDK does (because I do the timing in an ISR and not with polling which may get interrupted).
 
Peter
