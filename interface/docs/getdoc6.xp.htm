[snip]
 
>I FOUND THE REASON !!!
>
>The VxD name definition in .DEF file should be all CAPITAL CHARACTERS !!!
>
>I named my vxd as "MMCVxD" : the small "x" was the problem, shit!
  
Yes, this is a common reason for dynamic VxD unloading problems.
 
NuMega's DriverCentral Web site has an article
(http://www.numega.com/drivercentral/tech_tips/unloadvxd.shtml) that goes
over a few reasons for VxD unload problems.  The ALL CAPS name issue is one of things mentioned.
 
>
>This makes the VxD remains in memory even after calling CloseHandle().
>But still don't know why......
  
Before VXLDR (the system component responsible for managing dynamic VxDs) can unload a VxD, it must locate information it created when initially loading the VxD.  VXDLDR happens to do a case-sensitive search for this information using an ALL CAPS version of the VxD's name.  Thus, if a VxD somehow ends up with a mixed-cased name (as your's did because of the .DEF file you were using), this search will fail and VXDLDR won't be able to unload the VxD.
